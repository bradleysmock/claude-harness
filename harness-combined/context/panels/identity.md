## Identity, Authentication & Authorization Panel

*Active when identity-relevant code is in scope: OAuth 2.x / OIDC client or server code (`oauth`, `openid`, `passport`, `authlib`, `auth.js`, `next-auth`, `@auth0/`, `lucia`, `keycloak`, `hydra`, IdentityServer); JWT issuance or verification (`jsonwebtoken`, `jose`, `pyjwt`, `JJWT`); session management (cookie configuration, session stores, `express-session`, `iron-session`, `flask-login`, `django.contrib.auth`); password hashing (`bcrypt`, `argon2`, `scrypt`, `passlib`); MFA / WebAuthn / passkeys (`@simplewebauthn`, `py_webauthn`); authorization libraries (`oso`, `casbin`, `cerbos`, `opa`); SAML clients (`passport-saml`, `python3-saml`); identity-related routes (`/login`, `/oauth/callback`, `/.well-known/`, `/userinfo`, `/token`, `/authorize`). Generic HTTP correctness defers to HTTP/API; broad security concerns defer to Core; this panel covers the identity-specific machinery — OAuth/OIDC protocol correctness, token handling, session lifecycle, credential storage, and the authorization-enforcement-layer questions that are design-level rather than implementation-level.*

- **Aaron Parecki** — author of *OAuth 2.0 Simplified*; lead editor of the OAuth 2.1 spec; co-chair of the IETF OAuth working group; the canonical "what the spec says to do today" voice — Authorization Code + PKCE for everyone, no implicit flow, no resource owner password grant
- **Justin Richer** — lead author of *OAuth 2 in Action* (Manning); co-chair of IETF GNAP; one of the most experienced production OAuth/OIDC implementers; the canonical "what production OAuth actually looks like" voice — token introspection, scope design, multi-tenant identity, the operational tradeoffs

**Parecki's key positions:**

| Position | What it means in practice |
|----------|-----------------------------|
| **Authorization Code + PKCE is the answer for every client type** | OAuth 2.0 historically prescribed different flows by client type (implicit for SPAs, client credentials for machines, password grant for "trusted" apps). OAuth 2.1 collapses this: Authorization Code with PKCE is the only interactive flow. Implicit is deprecated; password grant is deprecated; PKCE is mandatory for public clients and recommended for confidential ones. Code shipping with implicit or password grant in 2025+ is on a deprecation list it will eventually be on the wrong side of. |
| **PKCE is not optional, and not just for SPAs** | The `code_verifier` / `code_challenge` exchange protects against authorization code interception across the entire redirect path — even confidential clients benefit, because the threat model includes proxies, redirect URI race conditions, and stolen logs. Public clients (SPAs, native apps, CLIs) cannot ship without it. |
| **Redirect URIs must be exact-match, not pattern-match** | Pattern-matched redirect URIs (`https://example.com/*`, prefix match, regex) have produced credential exfiltration in every system that's tried them. Exact string match against a registered allow-list is the requirement. Sub-path nesting, query-string variation, fragment-based identifiers — all open-redirect-equivalent in the OAuth context. |
| **`state` and `nonce` are not optional CSRF/replay parameters** | `state` on the authorization request, verified on callback, is the CSRF check for OAuth. `nonce` in OIDC requests, echoed in the ID token, is the replay check. Code that omits either, or accepts the callback without verifying them, has the security property of "any caller can complete the flow for any user." |
| **Stop the implicit flow, including in tutorials still recommending it** | OAuth implicit flow returned access tokens directly in the URL fragment. URL fragments leak through `Referer` headers, browser history, error pages, and CSP-violation reports. It was deprecated in 2020 (`oauth.net/2/grant-types/implicit/`). Code using `response_type=token` is on the breach roadmap. |
| **Bearer tokens belong in `Authorization` headers, never in URLs** | Access tokens in query strings (`?access_token=...`) end up in server logs, CDN logs, browser history, and the `Referer` of every outbound link from the page. Header-only carriage is the contract. The only exception (and even this is dispreferred) is downloadable assets where the header isn't an option — and there, signed short-lived URLs are the answer, not raw tokens. |

**Richer's key positions:**

| Position | What it means in practice |
|----------|-----------------------------|
| **JWT verification has more failure modes than its API suggests** | The `verify()` call must check: signature with the right key, `alg` against the allowed list (never `none`, never permissive), `iss`, `aud`, `exp`, `nbf`, `iat`. Libraries that take a key and "verify the token" without the caller specifying the algorithm have produced confusion attacks (RS256 token verified as HS256 using the public key as the HMAC secret) repeatedly. Pin the algorithm explicitly; pin the issuer; pin the audience. |
| **Opaque tokens + introspection beats stateless JWT for revocation** | JWT's "stateless verification" is exactly its weakness: a stolen JWT remains valid until expiry, with no revocation mechanism. For most server-side resource servers, opaque tokens checked against an introspection endpoint (RFC 7662) trade a network call for the ability to revoke immediately. Use JWTs at the edge where introspection round-trips are costly; use opaque tokens internally where revocation matters. |
| **Scopes are the authorization vocabulary; design them deliberately** | A token with `scope: 'admin'` granting everything is the OAuth equivalent of a root password. Design scopes per-resource and per-action (`orders:read`, `orders:write`, `users:read.basic`) so token compromise has a bounded blast radius. Scopes coarser than the resource server's actions are a design smell. |
| **Refresh tokens must rotate; absolute lifetime must be bounded** | Issuing a refresh token that grants new access tokens for years, with no rotation, no detection of replay, is the canonical breach vector. Rotate the refresh token on each use, detect reuse of an already-rotated token (compromise indicator), enforce an absolute lifetime ceiling separate from idle expiry. |
| **The ID token is not an access token** | OIDC ID tokens are *authentication artifacts* — proof of who the user is, intended for the client. Sending an ID token to a resource server as if it were an access token works on systems with permissive verification and is wrong on every level. Access token grants access; ID token identifies. Code that conflates them is broken; code that accepts both at a resource endpoint is dangerous. |
| **Identity at multi-tenancy is harder than authentication** | Authenticating "Alice" and authorizing "Alice can read this organization's data" are different problems. A correctly-signed token from your own IdP that grants access to *somebody*'s data is not authorization to access *this* data. Multi-tenant authorization belongs in the application layer (or in a policy layer like OPA/Cerbos), not in the token's `sub` claim. |
| **Federated logout is genuinely hard; plan it from the start** | Single sign-on without single sign-out is a half-built feature. Front-channel logout (iframe-based) is fragile; back-channel logout (server-to-server) requires every relying party to implement it correctly. Sessions outliving the IdP logout produce ghost sessions across the federation. |

*Synthesis:* Parecki evaluates whether the code follows the *current OAuth/OIDC specs* — Authorization Code with PKCE everywhere, no deprecated flows, exact redirect-URI matching, `state`/`nonce` verification, tokens never in URLs. Richer evaluates whether the code handles *the realities of production identity* — JWT verification done right (algorithm pinned, claims validated), token revocation strategy chosen deliberately, scopes designed for least-privilege, refresh-token rotation and replay detection, the ID-token-vs-access-token distinction respected, multi-tenancy enforced at the authorization layer rather than assumed from authentication. An identity stack can be spec-compliant and operationally fragile (correctly using PKCE while issuing year-long unrotated refresh tokens); or operationally hardened and spec-broken (refresh-token rotation done well but still shipping implicit flow). Both lenses are needed because authentication bugs are usually implementation bugs (patchable) while authorization bugs are usually design bugs (architectural — the BLOCKER class).

### Dimension 47: OAuth / OIDC Protocol & Token Handling
*Parecki, Richer*

| Hazard | What to look for |
|--------|-----------------|
| **Implicit flow in use** | `response_type=token` or `response_type=id_token token` in client code. Deprecated since 2020; replaced by Authorization Code with PKCE. Migration is mandatory before this becomes a removed feature in IdPs. |
| **Resource Owner Password Credentials grant** | `grant_type=password` collecting username/password directly. Deprecated; defeats the entire point of OAuth (delegated auth without sharing credentials). Migrate to Authorization Code + PKCE with a real authorization server UI. |
| **Authorization Code without PKCE** | Public client (SPA, native app, CLI) using Authorization Code flow without `code_challenge` / `code_verifier`. PKCE is mandatory for public clients; OAuth 2.1 mandates it for all clients including confidential. |
| **`client_secret` shipped in browser-executed code** | A JavaScript bundle or HTML page containing a `client_secret` literal. The "secret" is public the moment it ships. Public clients use PKCE, no secret. |
| **Redirect URI matched by prefix / wildcard / regex** | Authorization server (or client config) accepting redirect URIs by anything other than exact string match against a registered allow-list. Open-redirect-equivalent in the OAuth context; produces token exfiltration. |
| **`state` parameter missing or unverified** | Authorization request without `state`, or callback handler that doesn't verify the returned `state` against the value originally sent. CSRF on OAuth callbacks; an attacker can complete an OAuth flow for a victim. |
| **`nonce` parameter missing from OIDC** | OIDC authorization request without `nonce`, or callback handler that doesn't verify the `nonce` in the ID token matches the request. Replay attack vector. |
| **JWT `alg=none` accepted** | Verifier accepting tokens with `alg: 'none'` — no signature check. Library defaults usually reject this, but any code that explicitly enables `none` or uses a permissive `algorithms: []` list is broken. Pin allowed algorithms. |
| **JWT algorithm confusion** | Verifier accepting both RS256 and HS256 with the same key parameter — an attacker forges a token with `alg: 'HS256'` using the RSA *public* key as the HMAC secret. Pin a single algorithm (or a narrow allow-list); never derive the algorithm from the token header. |
| **JWT claims not validated** | `verify()` checking only signature and not `iss`, `aud`, `exp`, `nbf`, `iat`. A signed token from a different audience or issuer is still signed; missing `aud` check means a token meant for service A is accepted by service B. |
| **Token in query string** | Access token in `?access_token=...`. Logged everywhere, leaks via `Referer`, cached in browser history. Use `Authorization: Bearer ...`. |
| **No refresh-token rotation** | Long-lived refresh token that's accepted indefinitely without being rotated on each use. Stolen once, valid forever (until manual revocation). Rotate on use; detect reuse of an already-rotated token as a compromise indicator. |
| **Refresh tokens with no absolute lifetime** | Refresh token rotation alone isn't enough — chain renewal can extend a session indefinitely. Enforce an absolute lifetime ceiling (e.g., 30 days for web, 90 days for mobile) beyond which re-authentication is required. |
| **Refresh tokens in `localStorage`** | Tokens stored in `localStorage` are XSS-readable. Any third-party script (analytics, error tracking, ad library, supply-chain compromise) can exfiltrate them. Use `HttpOnly` cookies with `Secure` and `SameSite=Lax` (or stricter); for SPAs, use the backend-for-frontend pattern. |
| **ID token used as access token** | API endpoint accepting a token by checking signature/expiry without verifying it's an access token (vs. an ID token). ID tokens have different claims, different audiences, different lifetimes. Conflating them defeats the spec's separation. |
| **Scopes coarser than actions** | Single `admin` or `read` scope granting everything. Token compromise blast radius is the union of every action that scope allows. Design scopes per resource × action. |
| **Resource server skipping audience check** | Resource server accepting any signed token from the trusted IdP, regardless of `aud`. A token issued for service A is accepted by service B — accidental privilege escalation. Verify `aud` matches the resource server's identifier. |
| **Token introspection bypassed** | Opaque tokens validated by some local cache without calling the introspection endpoint, or with cached introspection results that outlive token revocation. Either don't cache, or cache for shorter than the worst-case revocation latency you accept. |
| **OIDC discovery endpoint trusted without TLS verification** | Client fetching `/.well-known/openid-configuration` over HTTPS but skipping certificate verification (`verify=False`, `rejectUnauthorized: false`). MITM rewrites the discovery document; you authenticate against the attacker. |
| **Authorization server allowing unregistered scopes** | `/authorize` accepting any scope string the client requests, with no allow-list per client. Compromised or misconfigured clients escalate scope. |
| **No token revocation endpoint / no logout invalidation** | Identity provider with no `/revoke` endpoint, or with `/logout` that doesn't invalidate the underlying session and tokens. "Logout" becomes a UI illusion. |

### Dimension 48: Session, Credential & Authorization Discipline
*Parecki, Richer*

| Hazard | What to look for |
|--------|-----------------|
| **Cookies missing `HttpOnly` / `Secure` / `SameSite`** | Session or auth cookies set without `HttpOnly` (XSS readable), without `Secure` (transmitted over HTTP), or without `SameSite=Lax` or stricter (CSRF surface). Each missing flag reopens a class of exploits the flag was designed to close. |
| **Session ID not regenerated on login** | Login flow that authenticates the user but keeps the same session identifier. Session fixation: attacker pre-supplies a session ID, victim logs in, attacker now shares the authenticated session. Regenerate on every privilege change. |
| **No idle timeout, no absolute timeout** | Sessions valid indefinitely until logout. Stolen session cookies remain valid forever; users on shared devices stay logged in past intent. Idle timeout (15-60 min for typical apps) plus absolute timeout (8-24 hours) is the baseline. |
| **Sessions stored client-side with weak integrity** | Signed cookies (JWT-in-cookie, signed session payloads) where the signing key is reused across environments, rotated rarely, or stored alongside the application code. Compromise of the signing key forges any session. |
| **Plaintext or weakly-hashed passwords** | Plaintext passwords (impossible in 2026, but still ships); MD5, SHA-1, unsalted SHA-256. Use Argon2id (preferred), bcrypt (cost factor ≥ 12 in 2026), or scrypt with parameters that match current hardware. |
| **No password breach checking** | New account or password-change flow that doesn't check the new password against breached-password databases (HIBP k-anonymous API, local copy of `pwnedpasswords`). Users continue rotating into already-compromised passwords. |
| **No MFA on privileged or money-handling accounts** | Admin accounts, accounts with payment access, accounts with billing/admin roles — no second factor required. Account compromise is single-factor. |
| **SMS as second factor for high-value accounts** | TOTP fallback to SMS, or SMS as primary second factor, for accounts where SIM-swap matters (financial, admin, identity provider). SMS is acceptable as a backup recovery channel, not as the primary second factor. |
| **TOTP without rate limiting** | TOTP verification endpoint accepting unlimited attempts. 6-digit codes are brute-forceable at sufficient attempt rate. Limit to a small number of attempts per code window. |
| **WebAuthn/passkeys absent where they'd be appropriate** | New auth flow built around passwords + TOTP for a user base where passkeys would substantially reduce phishing risk. WebAuthn / passkeys are the post-password story; consider for new flows. |
| **Password reset link with no expiration / no single-use** | Reset link valid for days, reusable. A leaked link (email forwarding, support ticket inclusion, screenshot) remains live. Expire in minutes to an hour; mark as consumed on first use. |
| **Account enumeration via error messages** | Login endpoint returning "user not found" vs. "wrong password," or password-reset endpoint confirming "we sent a link" only if the email exists. Uniform response prevents account enumeration. |
| **Account enumeration via timing** | Login endpoint that returns faster for non-existent users (no password hash compared). Constant-time response (always compute against a dummy hash for missing users) prevents the timing oracle. |
| **Authorization at controller but not at service** | Permission check in the HTTP route handler but not in the service or repository layer. Background jobs, admin tools, internal RPC paths, and refactors that move logic to a new caller all bypass the check. Defense in depth means every layer enforces. |
| **Authorization via client-supplied identifier (IDOR)** | `GET /orders/:id` returning the order without checking the requesting user owns it. The classic IDOR — incrementing IDs in the URL bar reveals other users' data. Authorization checks must consult the resource owner, not just the resource identifier. |
| **RBAC role granted that bundles unrelated permissions** | A single `support` role that grants read access to user data, billing data, and admin tools. Compromise of a support account is total. Split roles by responsibility; assign overlapping roles to people who need union access. |
| **Default-on permissions ("admin everything")** | New users provisioned with broad defaults; deprovisioning relies on remembering to remove. Default-deny with explicit grants is the discipline. |
| **No audit log on authorization decisions** | Permission grants, role changes, sensitive-resource accesses with no append-only audit trail. Forensics after incident is guesswork; compliance audits fail. |
| **Authorization caching without invalidation** | Role lookups, scope lookups, or permission decisions cached with no invalidation path. User loses access → cache still says they have access → effective revocation latency unbounded. Either don't cache, cache for short fixed windows, or invalidate on permission change events. |
| **Service-to-service trust by network position** | Internal API trusting requests because they came from inside the VPC. No mTLS, no signed tokens, no per-caller identity. One compromised service walks the whole mesh. Use mTLS or signed service tokens (SPIFFE/SPIRE pattern) for internal trust. |
| **Multi-tenant isolation by `WHERE tenant_id = ?` only** | Tenant isolation enforced purely in application SQL. One missing `WHERE` clause in one query leaks across tenants; one ORM call forgetting the scope is a data breach. Layer with row-level security in the database, separate schemas per tenant, or a query interceptor that injects the scope. |
| **Impersonation features without strict audit** | Admin "log in as user X" features used for support. Without scope (can do anything the user could do), audit (immutable record of every impersonated action), and consent (user notified, or impersonation permission granted), this becomes the backdoor that bypasses every other authorization check. |
| **Logout that doesn't invalidate server-side state** | "Logout" that clears client cookies but leaves the server session valid. Stolen cookies remain usable; "I logged out" gives the user a false guarantee. Invalidate on the server too. |
| **Permission logic scattered across the codebase** | `if user.role == 'admin' or user.id == resource.owner_id` patterns repeated across dozens of controllers, services, and templates. Refactoring is dangerous; consistency is unprovable. Centralize in a policy layer (OPA, Cerbos, Oso, Casbin, or an in-house module) the rest of the code consults. |

Parecki's design questions: for every OAuth flow this code initiates or handles, name the flow type, name where PKCE is verified, name where `state` is verified. If any of those answers is "I'm not sure," there's a security finding waiting. For every redirect URI in the configuration, is it exact-match against a registered allow-list? If wildcards or prefix matches are in use, the token exfiltration path is already open.

Richer's design questions: for every JWT this code verifies, what algorithm is pinned, which claims (`iss`, `aud`, `exp`, `nbf`) are checked, and what happens to a token that's valid by signature but stale or compromised? If revocation latency is "the token's `exp`," design that as a deliberate choice, not as a default. For every authorization decision this code makes, can a reader find the *single place* that decision is enforced? If the decision is enforced in three places and missing in a fourth, find the fourth.
