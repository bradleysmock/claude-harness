## React Panel

*Activation is governed by the trigger table in `skills/critique/SKILL.md`. TypeScript-level concerns defer to TypeScript/JS; generic UI / accessibility / Tailwind concerns defer to UI; this panel covers React-specific machinery — the render-as-description model, hooks rules and reactive closure semantics, Server Components, async/server state — and the patterns that distinguish "production React" from "tutorial React."*

- **Dan Abramov** — former React core team; the working voice on React's mental model — render as a pure description of UI for a given state, hooks as reactive primitives, Server Components as a separate runtime that composes with client components
- **Tanner Linsley** — TanStack Query / Router / Table / Form; canonical authority on the distinction between *server state* and *client state*, and on why `useEffect` for data fetching is the single most-recurring production bug class in React codebases

**Abramov's key positions:**

| Position | What it means in practice |
|----------|-----------------------------|
| **UI is a function of state** | A component describes what the UI should look like *for the current state*. You don't tell React what to do — you describe what should be true, and React reconciles. Code that tries to "do something when X changes" via imperative effects is usually fighting this model: the right question is "what does the UI look like when X is the current state?" |
| **Hooks are reactive primitives, not lifecycle hooks** | `useState`, `useEffect`, `useMemo`, `useCallback` participate in a render-driven reactivity system. The "rules of hooks" (top-level, same order, in components or hooks only) exist because React tracks them by *call index*, not name. Each render captures its own closure over state and props — code that reads "stale" values is reading the value from the render that scheduled it, which is correct, not buggy. |
| **`useEffect` is for synchronizing with external systems** | The right uses are: subscribing to a non-React data source, manually managing focus, integrating a third-party imperative library, firing analytics. The wrong uses are: deriving state from props (use `useMemo` or compute during render), fetching data (use a server-state library or Suspense + a framework loader), running code "after mount." Most `useEffect` calls in production React are this misuse. |
| **The component is the boundary; state is local by default** | "Lift state up" is real advice, but "keep state local until proven shared" is the prior. Premature lifting to context or a global store creates re-render cascades and tight coupling. Move state up when two siblings need to read it; not because "we might need it elsewhere later." |
| **Server Components are a separate runtime, not a perf tweak** | Server Components run only on the server, can be async, can read databases directly, and cannot use state, effects, refs, or browser APIs. The `"use client"` directive is the boundary marker. Mixing the two correctly means understanding the boundary; importing server-only code from a client component is the canonical compile error. |
| **Strict Mode runs effects twice in development on purpose** | The double invocation is a feature for surfacing effects that aren't idempotent. Code that breaks under Strict Mode usually had a latent bug — a subscription that doesn't clean up, an effect with side effects in render, a ref written during render. Suppressing Strict Mode is hiding the bug. |
| **Keys identify list items across renders** | The `key` prop tells the reconciler which element corresponds to which previous element. Using array index as a key for a dynamic list (items can be added, removed, or reordered) makes the reconciler match the wrong DOM nodes to the wrong items — focus jumps, inputs lose values, animations glitch. Use a stable per-item identifier. |
| **Hydration is rendering twice and asserting equality** | The server renders HTML; the client renders the same components and must produce the same output to attach handlers. Any non-deterministic input that differs across server and client (`Date.now()`, `Math.random()`, `window.innerWidth`, `localStorage`) causes a hydration mismatch. Either render the value only on the client (effect + state) or pass it explicitly from the server. |

**Linsley's key positions:**

| Position | What it means in practice |
|----------|-----------------------------|
| **Server state and client state are different problems** | Client state (form input, modal open/closed, selected tab) is owned by the client and changes only through user action. Server state (the list of orders, the user profile from the API) is *cached*, *asynchronous*, *stale by definition*, and not owned by the client at all. Treating them with the same tools (`useState`, Redux, Context) is the root cause of the most common React data bugs. |
| **`useEffect` + `fetch` is the canonical anti-pattern** | The naive `useEffect(() => { fetch(url).then(setData) }, [url])` lacks: deduplication of concurrent requests, cache, stale-while-revalidate, retry, abort on unmount, race-condition handling on rapid input, background refetch on focus, refetch on reconnect. Each one is a class of production bug. Use a server-state library (TanStack Query, SWR) or a framework loader (Remix, Next.js App Router). |
| **Query keys are the cache identity** | TanStack Query, SWR, and similar libraries identify cached data by a key. Two queries with the same key share cache; different keys are different cache entries. Unstable keys (new object literal each render, non-serializable values, missing dependencies) defeat the cache and produce loops of refetching. Stable, serializable, hierarchical keys (`['users', userId, 'orders']`) are the design. |
| **Mutations have a different lifecycle than queries** | A mutation is an imperative action with a discrete success/failure outcome. After it succeeds, the cached data it affected is stale — the mutation must invalidate (`queryClient.invalidateQueries`) or update (`setQueryData`) the affected query keys. Mutations without invalidation produce UI that shows stale data after writes. |
| **Optimistic updates require a rollback path** | "Show success immediately, sync to server in background" is a pattern that requires explicit rollback on failure: snapshot the prior cache, apply the optimistic update, on error restore the snapshot, on success let invalidation reconcile. Code that does the optimistic update but skips the rollback ships broken UX on every transient failure. |
| **Forms are usually server state with optimistic UI** | A form's submitted-and-saved state is server state. The form's in-progress editing state is client state. Conflating them — putting the entire edited form into a global store, or treating "saved" as a local boolean — is what produces the "lost edit" and "save button stuck" bugs in production forms. |
| **Race conditions on rapid input need cancellation** | A search box that fires a request per keystroke, where requests can return out of order, will display the wrong result for the displayed query. Either the data layer cancels in-flight requests on a new request (TanStack Query handles this via query key), or the fetch passes an `AbortSignal` from a per-key controller. Manual `setData` from `useEffect` does neither. |

*Synthesis:* Abramov evaluates whether the code respects React's render-as-description model — hooks used as reactive primitives, effects reserved for external-system synchronization, the client/server boundary in Server Components understood, hydration determinism preserved, reconciliation keys stable. Linsley evaluates whether *server state is treated as server state* — fetched through a cache layer with deduplication, retry, cancellation, and invalidation, not through ad-hoc `useEffect + setState` patterns that re-invent (badly) what TanStack Query / SWR / framework loaders already solve. Most "weird React bugs" in production are one of: a `useEffect` that should have been derived state, a hydration mismatch from non-deterministic render input, a stale closure from a missing dependency, or `useEffect` doing the job of a server-state library.

## Review Dimensions

---

### Dimension 37: React Mental Model & Hooks Correctness
*Abramov*

| Hazard | What to look for |
|--------|-----------------|
| **`useEffect` for derived state** | `useEffect(() => setFullName(`${first} ${last}`), [first, last])` — derived value stored in state with an effect that copies it across. Compute during render: `const fullName = `${first} ${last}`` (or `useMemo` if the calculation is expensive). |
| **`useEffect` for data fetching** | `useEffect(() => { fetch(...).then(setData) }, [...])` in component code. Punt to Dimension 38 (server-state) — but flag here because the symptom presents as a React hooks problem. |
| **Missing effect dependency** | `useEffect(() => { doThing(x, y) }, [x])` — `y` read inside, omitted from deps. ESLint plugin catches most; flag any `// eslint-disable-next-line react-hooks/exhaustive-deps` without a same-line explanation. |
| **Stale closure trap** | An interval, subscription, or async callback set up in an effect captures the props/state from that render. The "current" props in the running callback are months stale by user time. Use a ref to hold the latest value if it must be read inside a long-lived callback. |
| **Conditional hook call** | `if (cond) useState(...)`, `if (cond) useEffect(...)` — violates rules of hooks. React tracks hooks by call index; conditional calls misalign the index. |
| **State update during render** | `if (someCondition) setSomething(value)` at the top of a function component — fires on every render, schedules another, infinite loop. The exception is the documented "deriving state from props" pattern (snapshot the prop, only call setState when the snapshot diverges). |
| **`useState` for derived state** | `const [fullName, setFullName] = useState(...)` then an effect to keep it in sync. Compute during render or use `useMemo`. The local state shadows the prop and drifts. |
| **Array index as key on dynamic list** | `items.map((item, i) => <Row key={i} ... />)` where items can be inserted, removed, or reordered. Use a stable per-item identifier. |
| **New object/array/function in render passed to memoized child** | `<MemoChild config={{ a: 1 }} />` — `config` is a new reference every render; `React.memo` is defeated. Use `useMemo` for the value, `useCallback` for the function, or restructure to pass primitives. |
| **`useMemo` / `useCallback` everywhere** | Defensive memoization of every prop and function with no measurement. The memoization comparison itself has cost; if the child isn't `React.memo`'d and the work is cheap, it's net-negative. Memoize where measurement shows benefit. |
| **`useRef` written during render** | `ref.current = computeSomething()` at the top of a function body. Refs are for values that persist across renders and *don't* drive the UI; writing one during render couples render to the side effect and breaks under Strict Mode's double invocation. |
| **Hydration mismatch source** | `Date.now()`, `Math.random()`, `new Date().toLocaleString()`, `window.X`, `localStorage`, `navigator.X` read at the top of a component that also renders on the server. Causes "text content did not match" warnings; production loses the entire hydrated subtree. Read on client only (`useEffect` + state). |
| **Server Component / Client Component boundary violation** | A Server Component importing a Client Component is fine; a Client Component importing server-only code (database client, server-side secret, Node-only API) is a compile or runtime error. Check `"use client"` placement and the import graph crossing it. |
| **`"use client"` on a leaf when the parent could be marked** | Marking too many components as client when the boundary could be drawn higher — pulls the whole subtree to the client and forfeits Server Component benefits. The directive marks a *boundary*; place it deliberately. |
| **Missing Suspense boundary around async** | An async Server Component or a Suspense-enabled query without a `<Suspense>` boundary ancestor. The async unit suspends the closest ancestor — if none, the whole route falls back to the loading screen of the root. Place boundaries at granularity that matches the UX. |
| **Missing Error Boundary** | No Error Boundary surrounding components that can throw (especially Suspense-thrown promises rejecting, query errors, render errors in untrusted data). Unhandled render error unmounts the whole tree to the nearest boundary; if none, the app blanks. |
| **`useLayoutEffect` where `useEffect` would do (and vice versa)** | `useLayoutEffect` runs synchronously before paint — blocks rendering, breaks SSR (warns). Reserve for measurements / DOM mutations that must be invisible. Effects that don't read layout should use `useEffect`. |
| **Direct DOM manipulation in render** | `document.getElementById(...)` or DOM mutation in component bodies. Breaks SSR, defeats reconciliation, conflicts with React's lifecycle. Use refs and effects. |
| **Children mutation** | `props.children.map(...)` then mutating the result, or `React.Children.map` with the assumption children stay stable. Children are a description, not a mutable list. |
| **Context for everything** | Wrapping the app in a dozen context providers used by leaf components. Every consumer re-renders on any context value change. Group context by change cadence; use a state-management library or `useSyncExternalStore` for high-frequency state. |
| **Forgetting `useReducer` for coupled state** | Multiple `useState` calls whose updates always happen together and depend on prior state. A reducer expresses the transition as data; the component reads simpler. |

### Dimension 38: Server State & Data Fetching
*Linsley*

| Hazard | What to look for |
|--------|-----------------|
| **`useEffect` + `fetch` for data loading** | Component code containing `useEffect(() => { fetch(url).then(...).then(setData) }, [...])`. Missing: dedup, cache, stale-while-revalidate, retry, abort, focus refetch, reconnect refetch, race-condition handling. Use TanStack Query / SWR, a framework loader, or Suspense + a data library. |
| **Manual cache via `useState` + `Context`** | A "store" component holding API responses in `useState` and providing them via Context, manually invalidating on mutation. Reinventing a server-state library, badly. The reinvention is missing most of the discipline and ships with bugs the library has fixed. |
| **Loading state in `useState`** | `const [loading, setLoading] = useState(false)` flipped manually around fetches. Two requests in flight produce inconsistent state. The query library tracks `isLoading` / `isPending` / `isFetching` per query. |
| **Unstable query key** | `useQuery({ queryKey: ['users', { filter }], ... })` where `filter` is a new object each render → new key each render → cache miss every render → request loop. Keys must be stable or memoized; prefer primitive components in keys. |
| **Non-serializable value in query key** | A function, Date, or class instance in the query key. The cache uses structural comparison; non-serializable values fail equality unpredictably. |
| **Missing mutation invalidation** | `mutate()` to update a record, then the UI continues showing the cached pre-mutation data. The mutation handler must call `queryClient.invalidateQueries({ queryKey: [...] })` or `setQueryData` to update the cache. |
| **Optimistic update without rollback** | A mutation that calls `setQueryData` to show the new state immediately, with no `onMutate` snapshot and no `onError` restoration. Transient failure leaves the UI lying about the persisted state. |
| **Pagination refetching all pages** | A page-changing UI keying queries on `['items', page]` instead of using `useInfiniteQuery` or `keepPreviousData` — every page click flashes loading state and refetches earlier pages on navigation back. |
| **Infinite query without bounded retention** | `useInfiniteQuery` accumulating pages with no `maxPages` / no eviction. Long-running view loads thousands of pages into memory. |
| **SSR cache not hydrated** | Server-rendered page that pre-fetches data with the query library, but the client mounts without dehydrating/hydrating the cache (`Hydrate`/`HydrationBoundary`) — client refetches everything on hydration, defeating the SSR cost. |
| **Race condition from manual fetch** | Search input firing a request per keystroke with `useEffect` + `fetch`. Earlier slow requests resolve after later fast ones; UI shows results for an earlier query. Either use a keyed query library (cancellation is automatic) or pass `AbortSignal` and track per-input controllers. |
| **No `AbortSignal` propagation to `fetch`** | A `fetch` call inside a query function or effect without forwarding the abort signal the caller provides. Cancellation requested → request continues; result of cancelled work overwrites later state. |
| **Polling via `setInterval`** | `useEffect(() => { const id = setInterval(refetch, 5000); return () => clearInterval(id) }, [])` — works but reinvents `refetchInterval`, doesn't pause on tab hidden, doesn't dedup with focus refetch. |
| **Retry on non-retryable errors** | Query retry config (the default is several retries) firing against 4xx errors — never going to succeed, wastes attempts, delays the error surfacing to the user. Distinguish retryable (network, 5xx) from terminal (4xx, validation). |
| **Server state in Redux / Zustand** | A global client-state store holding API response data. Every cross-cutting concern (caching, invalidation, retries, dedup) is now your problem. Server state libraries solve it; pick the right tool. |
| **Form state in server-state cache** | The in-progress edited form values stored as a query result and `setQueryData` on each keystroke. Cache invalidation on focus / reconnect blows away the user's edits. Form-while-editing is client state; submit produces a mutation. |
| **No suspense / no loading affordance** | Async component or query with neither a Suspense boundary nor a `query.isLoading` branch. Renders empty, then jumps to data — looks broken. |
| **Optimistic UI without conflict handling** | Optimistic update assumes the server accepts; conflict (validation, race with another client, permission change) leaves the optimistic state visually but unsaved. Build a conflict-resolution UI or fall back to pessimistic for high-stakes mutations. |

Abramov's design questions: for every `useEffect` in this code, can you state which external system it synchronizes with? If the answer is "it derives state" or "it fetches data," the effect is in the wrong place. For every component that renders on both server and client, does it produce the same output given the same props? If not, you have a hydration mismatch waiting to fire.

Linsley's design questions: for every piece of data your UI displays, can you classify it as client state or server state? For each piece of server state, what is its cache key, what invalidates it, and what shows during refetch? If your answer involves `useEffect`, `useState`, and a `fetch` call, you've reinvented the problem; the library exists.
