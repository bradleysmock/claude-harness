## SolidJS Panel

*Active when SolidJS is in the project: `solid-js` or `@solidjs/start` in `package.json`; `**/*.{jsx,tsx}` files using Solid's JSX runtime (`vite.config.*` with `vite-plugin-solid`); `app.config.{js,ts}` (SolidStart); or routes under `src/routes/**` matching SolidStart conventions. TypeScript-level concerns defer to TypeScript/JS; generic UI concerns defer to UI; this panel covers Solid-specific machinery — the fine-grained signal-based reactivity, the run-once component model, JSX-as-direct-DOM-compilation, and the SolidStart boundary between server and client.*

- **Ryan Carniato** — SolidJS creator; SolidStart lead; the canonical voice on fine-grained reactivity, the "components run once" model, and why JSX compiled to direct DOM operations (no virtual DOM) is a different framework even when the source code looks like React

Solid is a single-panelist panel by design — like Svelte, it intentionally compresses surface area. The framework's design proposition is "React's mental model with Svelte's performance," but the most common source of production Solid bugs is exactly this surface similarity: developers write Solid components with React habits (destructuring props, running effects expecting re-render, calling `array.map` over reactive lists) and the code compiles cleanly but doesn't react. The hazard set is narrow and concentrated.

**Ryan Carniato's key positions:**

| Position | What it means in practice |
|----------|-----------------------------|
| **Components run once; the body is setup, not render** | A Solid component function executes *one time per instance* to set up the reactive graph. There is no re-render. Code in the function body that expects to run again when a signal changes is built on a React mental model that doesn't apply. Code that *reads* a signal in JSX participates in the reactive graph; code that runs once at component setup does not. |
| **Signals are the primitive; call them to read, invoke setter to write** | `const [count, setCount] = createSignal(0)` — `count` is a *function* you call to subscribe + read; `setCount(n)` or `setCount(c => c + 1)` writes. The function-call surface is what enables fine-grained tracking: the compiler/runtime knows the call site, so it can subscribe just that expression. `count` (as a value) and `count()` (as a read) are different things; mixing them up is the canonical first-week Solid bug. |
| **Reactivity is fine-grained at the expression level** | When a signal changes, only the specific DOM expressions that read it re-evaluate — not the surrounding component, not even the surrounding JSX element. `<div>{count()}</div>` updates the text node, not the div. This is what makes Solid fast; it's also why the component body running once isn't a limitation — only the reactive expressions need to re-run. |
| **Destructuring props loses reactivity** | `props` is a reactive proxy. `function Foo({ name })` destructures at call time, captures the initial value, and never tracks updates. Use `props.name` (tracked) or `splitProps` / `mergeProps` for explicit destructure. Same hazard family as Vue's destructured-`reactive()`; the diagnostic and fix shape are similar. |
| **`<For>`, `<Show>`, `<Switch>` over native control flow** | `items().map(item => <Row {...item} />)` re-creates every row when the array reference changes — every render of every row, no keying. `<For each={items()}>{item => <Row {...item} />}</For>` does fine-grained reconciliation: only added/removed/moved items mutate the DOM. Same applies to `<Show>` over `&&`/ternary for conditional blocks and `<Switch>` over `switch` JSX expressions. |
| **`createEffect` for side effects; `createMemo` for derivation** | Same discipline as React's `useMemo`/`useEffect` and Vue's `computed`/`watch`. `createMemo(() => expensive(deps()))` produces a tracked, cached derived signal. `createEffect(() => fn(deps()))` runs `fn` on changes for side effects. Derivations inside effects with manual signal writes are usually memos written awkwardly. |
| **`createStore` for nested mutable state** | `createStore({...})` returns a deep reactive proxy with path-based tracking — mutating `store.user.name = 'x'` triggers only consumers that read `store.user.name`. Use stores for nested state where granular updates matter; use signals for scalars and for state where the whole-value identity is the unit. |
| **`onMount` and `onCleanup` are the lifecycle** | No `useEffect` / `componentDidMount` / `onMounted` analog with reactive tracking — `createEffect` is the reactive primitive, `onMount` is the imperative-lifecycle one (no tracking, runs once). For cleanup of any subscription / timer / resource, return a cleanup function from a `createEffect` or call `onCleanup(...)` inside the setup or effect. |
| **`batch` groups updates into a single reactive notification** | `batch(() => { setA(1); setB(2); setC(3) })` triggers downstream effects/memos once with all three new values. Unbatched, three separate notifications can produce two intermediate states that effects observe and react to. |
| **`untrack` is the escape valve, not the rule** | Reading a signal inside an effect/memo subscribes the effect to it. `untrack(() => someSignal())` reads without subscribing — for cases where you need the current value but don't want updates to retrigger. Reaching for it routinely is a sign the dependency graph is wrong. |
| **SolidStart's `"use server"` and route loaders are the data boundary** | SolidStart provides route-level data via `createAsync` + `cache(...)` and server functions via `"use server"` — the canonical fetch-on-the-server-and-stream-down pattern. `createResource` in component code for data that should be route-level forfeits the framework's pre-rendering, streaming, and invalidation. |

*Synthesis:* Carniato evaluates whether the code honors Solid's run-once, signal-tracking model — props read through the proxy not destructured, control flow expressed with `<For>`/`<Show>`/`<Switch>` not native JS, effects reserved for side effects with `createMemo` for derivation, batches around grouped updates, signal calls (`count()`) distinguished from references (`count`), and SolidStart's loader/server-function boundary used for data rather than `onMount` + `fetch`. The bug pattern that defines Solid in production isn't a single class of error — it's the React-developer-writing-Solid pattern: destructured props, `.map` over arrays, effects expecting re-render. Catching this pattern is what the panel is for.

## Review Dimensions

### Dimension 43: SolidJS Reactivity, Components & SolidStart
*Carniato*

| Hazard | What to look for |
|--------|-----------------|
| **Destructured props in component signature** | `function Card({ name, count }) { ... }` — `name` and `count` are snapshots, not tracked reads. Use `props.name` / `props.count`, or `splitProps(props, ['name'])` for partial destructure with reactivity preserved. The canonical first-week-of-Solid bug. |
| **`array.map()` over a reactive list** | `<>{items().map(item => <Row {...item} />)}</>` — recreates every row on every change of `items()`. Use `<For each={items()}>{item => <Row {...item} />}</For>` for keyed reconciliation. For index-based reconciliation (rare), `<Index>`. |
| **Native `&&` / ternary for conditional blocks** | `{showHeader() && <Header />}` re-evaluates the whole expression and mounts/unmounts `<Header />` on each toggle. `<Show when={showHeader()}><Header /></Show>` is the fine-grained equivalent — also gives `fallback` slot. |
| **`switch (x()) { ... }` returning JSX** | Switch-statement JSX inside expressions. `<Switch>` + `<Match when={...}>` participates in fine-grained tracking. |
| **Reading signal without calling it** | `<div>{count}</div>` where `count` is the signal function — renders `function count() { ... }`. Should be `<div>{count()}</div>`. JSX displays whatever value is passed; the function isn't auto-invoked. |
| **Calling a setter as a read** | `setCount(prev => prev + 1)` is correct; `setCount()` with no argument returns the current value but is unidiomatic — use `count()` to read. Reading via the setter signals confusion about the read/write split. |
| **`createEffect` reading a value not a signal** | `createEffect(() => doStuff(props.foo))` works (reactive read on `props.foo`); `createEffect(() => doStuff(staticValue))` runs once and never again — the effect has nothing to track. If "react to a change" was expected, the value isn't a signal. |
| **Effect cleanup missing** | `createEffect(() => { const id = setInterval(...); /* no return */ })` — interval leaks across signal changes that re-run the effect. Return a cleanup function from the effect, or use `onCleanup(...)` inside it. |
| **`createMemo` for trivial derivation** | `const doubled = createMemo(() => count() * 2)` for a one-line cheap calc used in one place. Memo has overhead; an inline `{count() * 2}` in JSX is fine. Memos earn their cost when (a) the derivation is expensive or (b) multiple downstream consumers benefit from sharing. |
| **`createMemo` with side effects** | A memo that calls setters or runs side-effect calls in its function body. Memos must be pure. Side effects belong in `createEffect`. |
| **Effect cycle (effect writes a signal it reads)** | `createEffect(() => { setX(y() + 1) })` where another effect reads `x()` and sets `y()`. Infinite loop, sometimes caught by Solid as a cycle, sometimes manifesting as runaway memory/CPU. |
| **Unbatched grouped updates** | Multiple `setX(...); setY(...); setZ(...)` in a row outside a `batch()` block. Effects/memos that depend on more than one of them observe intermediate states; downstream consumers re-run more than necessary. |
| **`createStore` replaced wholesale instead of updated via path** | `setStore(newStoreObj)` replacing the whole store, or `setStore('list', [...newList])` when path-based updates would preserve fine-grained tracking. Use `setStore('list', i, 'name', newName)` or `produce(...)` for nested mutations. |
| **Reactivity boundary violated by accessing a signal outside a reactive context** | Reading a signal in a plain function called during setup (or in `onMount`, which is non-reactive) and storing the result — the result is a snapshot. Wrap in `createEffect` or `createMemo` if reactivity is needed. |
| **`onMount` for data fetching in a SolidStart route** | `onMount(async () => { setData(await fetch(...)) })` in a route component instead of a route loader (`cache(...)` + `createAsync(...)`) or a server function (`"use server"`). Forfeits SSR, prefetching, and the framework's revalidation. |
| **Server function returning non-serializable data** | A `"use server"` function returning a Date, Map, Set, class instance, or function — SolidStart serializes the result for the client; non-serializable values become unusable on the other side. Project to primitives at the boundary. |
| **Hydration mismatch source** | `Date.now()`, `Math.random()`, `window.X`, `localStorage` read in component setup where the component also renders on the server. Solid's hydration matching expects identical output. Read on the client only (signal + effect, or `<ClientOnly>` boundary). |
| **`createResource` in a component when route-level data is what's needed** | `createResource(() => fetch(url))` in a leaf component for data the entire route depends on — the resource starts loading after mount, after SSR. Hoist to a route loader for pre-rendered, streamable data. |
| **`<Suspense>` missing around async** | A `createResource` or async server function consumer rendered without a `<Suspense>` ancestor. The fallback bubbles up to the nearest one; if none, the whole route shows the root fallback. |
| **`<ErrorBoundary>` missing** | No `<ErrorBoundary>` around components that may throw — including resource errors propagating through Suspense. Unhandled error unmounts to the nearest boundary; without one, the app blanks. |
| **Refs treated as React-style ref objects** | `let myRef!: HTMLDivElement; <div ref={myRef} />` — Solid refs are *callback functions* the compiler invokes with the element, or simple variable assignments. Code copying React's `useRef().current` model produces type errors and stale references. |
| **`createSignal` for component-mounted-once state when `let` would do** | `const [name] = createSignal('static')` where the value never changes. A plain `const name = 'static'` is fine; signals are for reactive state. Defensive over-use creates noise. |
| **JSX expression evaluated outside the reactive scope** | An object/array literal computed in setup and passed to a child, where the expectation was for it to update on signal change. Setup runs once; the literal is frozen. Wrap in a function (`prop={() => buildObj(signal())}`) so the child can call it reactively. |

Carniato's design questions: for every component in this code, what runs once vs. what runs on signal changes? If you're not sure, the mental model isn't loaded — and "I'll add `createEffect`" is usually the wrong reflex. For every `array.map` and `&&`/ternary in JSX, is the surrounding control flow fine-grained (`<For>`, `<Show>`) or whole-expression? For every prop read in a component body, is it `props.x` or a destructured `x`? Solid bugs concentrate at these three boundaries.
