## Applied Cryptography Panel

*Active when cryptographic code is in scope: files importing crypto libraries — Python (`cryptography`, `pycryptodome`, `nacl`, `hashlib` for security purposes, `hmac`, `secrets`), Node (`crypto`, `noble-*`, `tweetnacl`, `libsodium-wrappers`), Go (`crypto/*`, `golang.org/x/crypto`, `filippo.io/age`), Java/Kotlin (`javax.crypto`, `java.security`, `BouncyCastle`, `Tink`), Rust (`ring`, `rustls`, `aes-gcm`, `chacha20poly1305`, `sodiumoxide`), Swift (`CryptoKit`); JWT signing/verification libraries; password-hashing libraries (`bcrypt`, `argon2`, `scrypt`); TLS configuration code; signing/verification code (`ed25519`, `ecdsa`, signing payloads); files constructing or implementing protocols that touch cryptographic primitives. Identity-protocol-level concerns (OAuth, OIDC, JWT-as-bearer-token usage) defer to the Identity panel; this panel covers the cryptographic primitives, algorithm choices, and protocol-construction discipline that sit underneath.*

- **Filippo Valsorda** — long-time Go cryptography maintainer; creator of `age`, `mkcert`, and the canonical post-quantum Go libraries; the working voice on cryptographic engineering as systems engineering — "crypto should be boring," high-level APIs over primitives, constant-time as a default discipline rather than a special-case concern
- **Matthew Green** — cryptographer at Johns Hopkins; *A Few Thoughts on Cryptographic Engineering* (blog); the working voice on protocol failure analysis — why real-world cryptographic schemes break, the gap between "the math is sound" and "this specific implementation is exploitable," and the cost of getting any of it wrong

**Valsorda's key positions:**

| Position | What it means in practice |
|----------|-----------------------------|
| **Crypto should be boring** | The right answer for most "I need to encrypt this" questions is a high-level library — `age` for files, `libsodium` / NaCl for application crypto, TLS for transport. Reaching for AES, RSA, or ChaCha20 directly puts you in the role of designing a cryptographic protocol — a role that has consistently produced the most-exploited bugs in the field, even when written by experts. "I just need raw AES-CBC" is rarely true and almost always the start of a vulnerability. |
| **Constant-time discipline is non-negotiable** | Any comparison involving a secret (HMAC, password hash, signature, capability token) must be constant-time — `hmac.compare_digest`, `crypto.timingSafeEqual`, `subtle.ConstantTimeCompare`. Plain `==` short-circuits on the first byte mismatch; the timing difference is a side channel an attacker can read across a network. This applies to every secret comparison, every byte of every signature check. |
| **AEAD is the only encryption** | Modern encryption is *authenticated* encryption (AES-GCM, ChaCha20-Poly1305, XChaCha20-Poly1305). Unauthenticated encryption (AES-CBC alone, AES-CTR alone, raw stream ciphers) gives an attacker the ability to modify ciphertext in ways the recipient cannot detect — padding oracles, bit-flipping attacks, malleability. If your code uses an unauthenticated mode, it has integrity bugs whether you've noticed them yet or not. |
| **Nonce reuse is catastrophic; design it out** | AES-GCM, ChaCha20-Poly1305, AES-CTR, and most modern modes lose all security guarantees if a nonce is reused with the same key. "Catastrophic" here is literal: nonce reuse in GCM recovers the authentication key, allowing arbitrary forgeries. Generate nonces from a CSPRNG (96-bit minimum); use XChaCha20-Poly1305 if random-nonce safety margin matters; never derive nonces from message content or counter alone without a discipline that proves uniqueness. |
| **`math/rand` is not for security; `crypto/rand` is** | Every language has a fast PRNG (`Math.random`, `random.random`, `math/rand`, `rand()`) and a cryptographic one (`crypto.randomBytes`, `secrets.token_bytes`, `crypto/rand`, `getrandom`). Using the fast one for tokens, session IDs, IVs, keys, or salts produces predictable values an attacker can guess. The distinction is the most-mixed-up thing in production cryptographic code. |
| **High-level APIs hide the failure modes for a reason** | When you call `libsodium.crypto_secretbox_easy` or `age` encryption, the library is choosing the cipher, the nonce-generation strategy, the authenticated-encryption pairing, the key-derivation step. Those are exactly the choices most often gotten wrong by hand. Stay at the level where the library makes the choices for you — descending to primitives means accepting responsibility for choices the high-level API was making correctly. |

**Green's key positions:**

| Position | What it means in practice |
|----------|-----------------------------|
| **The primitive is sound; the protocol is where it breaks** | AES is not broken. RSA-OAEP is not broken. ChaCha20 is not broken. The cryptographic breaks of the last two decades have almost all been protocol-level: padding oracles in CBC, Bleichenbacher attacks on PKCS#1 v1.5, downgrade attacks (FREAK, LOGJAM, POODLE), nonce reuse, algorithm confusion (RS256 vs HS256 in JWT). Reviewing crypto code means reviewing the protocol the primitives are wired into, not just the primitives. |
| **Domain separation prevents cross-protocol attack** | When the same key is used to sign messages of multiple types — login challenges, password-reset tokens, application data — a signature on one type can be replayed as a signature on another. Domain separation (prefixing the signed data with a context string, deriving sub-keys per use) is the fix. Code that signs raw payloads with no context tag is one cross-protocol-reuse vulnerability away from compromise. |
| **Padding oracles, downgrade attacks, and side channels are real** | A decryption routine that returns "invalid padding" vs. "invalid MAC" gives the attacker a padding oracle (Lucky 13, POODLE family). A protocol that accepts a downgrade to a weaker cipher gives the attacker FREAK. Constant-time comparison and uniform error handling aren't paranoia — they're closing channels attackers actively probe. |
| **Algorithm agility without governance becomes algorithm confusion** | Supporting multiple algorithms ("we accept RS256 and HS256") sounds flexible until an attacker chooses which one verification uses. JWT's `alg` field is the textbook case. Either pin to a single algorithm at the boundary or implement a strict allow-list with no caller control over selection. |
| **Forward secrecy is what protects past traffic from future key compromise** | Ephemeral key agreement (ECDHE, X25519) means that even if the long-term signing key is later compromised, prior sessions remain confidential. Protocols using only static keys (or RSA key transport, which lacks forward secrecy) trade today's convenience for tomorrow's bulk decryption of every session ever recorded. The migration cost was real fifteen years ago; today it's a default. |
| **Post-quantum is no longer a future concern for long-lived secrets** | NIST's PQC standards landed in 2024 (ML-KEM, ML-DSA, SLH-DSA). Data encrypted today with classical primitives — RSA, ECDH, ECDSA — is recoverable if a sufficient quantum computer is built within the secret's relevant lifetime. For ephemeral session keys this is moot; for backups, archives, signed firmware, and any data with a multi-decade sensitivity horizon, hybrid (classical + PQC) is the responsible default for new designs. |
| **Cryptography is failure-prone in ways code review cannot catch** | A code reviewer can verify that `bcrypt` is used and the cost factor is sensible. A reviewer cannot easily catch: a timing side channel that requires statistical analysis to detect, a flawed nonce generator that produces low-entropy outputs under load, a key reuse that only manifests at certain data volumes. The mitigation is using well-audited libraries (Tink, libsodium, ring, BoringSSL) rather than glue-code over primitives — not heroic review. |

*Synthesis:* Valsorda evaluates whether the code is built on *boring, high-level cryptographic engineering* — AEAD over unauthenticated encryption, CSPRNG over fast PRNG, constant-time everywhere a secret is compared, high-level APIs (`libsodium`, `age`, Tink) over primitive composition. Green evaluates whether the *protocol layer* is sound — domain separation between cryptographic uses of the same key, no padding oracles, no algorithm confusion, no downgrade paths, forward secrecy where past-traffic confidentiality matters, post-quantum considered for data with relevant lifetime. A cryptosystem can be primitive-correct (right algorithms, right libraries) but protocol-broken (no domain separation, downgrade-accepting); or protocol-sound but primitive-misused (correct flow with `Math.random()`-derived nonces). The two lenses are not interchangeable, and reviewers untrained in cryptography should be conservative: when in doubt, **call it out and route to a specialist** — cryptographic findings have a higher false-negative cost than most.

## Review Dimensions

### Dimension 49: Primitives, Algorithms & Construction
*Valsorda, Green*

| Hazard | What to look for |
|--------|-----------------|
| **MD5 or SHA-1 used for security** | MD5 for integrity, signatures, certificate hashing, identifiers-with-collision-relevance; SHA-1 the same. Both have practical collisions. Use SHA-256, SHA-3, or BLAKE2/BLAKE3 for new code. (HMAC-SHA1 remains acceptable for legacy interop but isn't a default for new design.) |
| **ECB mode anywhere** | `AES-ECB`, `Cipher.getInstance("AES")` (which defaults to ECB on many JCE providers), or any encryption that processes blocks independently. ECB leaks structure — the same plaintext block produces the same ciphertext block. Famous "encrypted Tux" demonstrations. Use an AEAD mode. |
| **Unauthenticated encryption (CBC, CTR, raw stream)** | `AES-CBC`, `AES-CTR`, `ChaCha20` used without an accompanying MAC, or with hand-rolled encrypt-then-MAC. Lacks integrity; produces padding-oracle and bit-flipping vulnerabilities. Use AEAD (AES-GCM, ChaCha20-Poly1305, XChaCha20-Poly1305). |
| **AEAD tag not verified before using plaintext** | Decryption code that returns plaintext on a tag-verification failure, or uses partial plaintext before tag verification completes. The whole point of AEAD is to gate the plaintext on integrity. |
| **Nonce / IV reuse** | Same nonce used twice with the same key in AES-GCM, AES-CTR, ChaCha20-Poly1305 — catastrophic for both confidentiality (CTR) and integrity (GCM authentication key recovery). Look for: counters initialized from short hashes, fixed nonces stored as constants, nonces derived from message identifiers without per-key uniqueness guarantee. |
| **Nonce / IV from non-CSPRNG source** | Nonce generated with `Math.random()`, `time.Now().UnixNano()`, or a hash of the plaintext. Use `crypto.randomBytes` / `secrets.token_bytes` / `crypto/rand`. For high-volume use, XChaCha20-Poly1305 (192-bit nonces) tolerates random nonces safely; AES-GCM's 96-bit nonce space is at risk past 2^32 messages per key. |
| **Insecure RNG for tokens / keys / salts / IDs** | `Math.random()`, `random.random()`, `rand()`, `math/rand` used to generate session IDs, password-reset tokens, API keys, salts, IVs, or any value that must be unpredictable. Use the language's CSPRNG: `crypto.randomBytes` (Node), `secrets.token_*` (Python), `crypto/rand` (Go), `SecureRandom` (Java), `OsRng` (Rust). |
| **Plain `==` comparison of secrets / signatures / MACs** | Comparison of two byte strings where one is attacker-supplied and the other is a secret or derived from one. Short-circuit timing leaks the position of the first mismatching byte. Use `hmac.compare_digest`, `crypto.timingSafeEqual`, `subtle.ConstantTimeCompare`, `MessageDigest.isEqual` (Java), `ring::constant_time::verify_slices_are_equal` (Rust). |
| **Raw password used as encryption / signing key** | `key = password.encode()` or `key = sha256(password)` with no KDF. Passwords have far less entropy than cryptographic keys; using them directly as keys makes offline brute-force trivial. Run through a password-based KDF (Argon2id, scrypt, PBKDF2) sized to current hardware. |
| **Weak password-hashing parameters** | bcrypt with cost factor < 12 (2026 floor); PBKDF2-HMAC-SHA256 with < 600,000 iterations; Argon2id with memory < 64 MiB or time cost < 3. Parameters need to scale with hardware; pinned values from a 2018 blog post are now too weak. |
| **No salt, or salt reused across users** | Password hash function applied without a per-user random salt. Identical passwords hash identically; rainbow tables apply. Salt must be random, per-record, stored alongside the hash. |
| **RSA-PKCS#1 v1.5 padding for encryption** | RSA encryption using PKCS#1 v1.5 padding — Bleichenbacher's attack (1998, still finding new variants in 2024). Use RSA-OAEP for encryption, RSA-PSS for signatures, or move to elliptic-curve schemes (X25519 + ChaCha20-Poly1305 via NaCl box for encryption; Ed25519 for signatures). |
| **RSA key size < 2048 bits** | RSA-1024 is broken in well-funded threat models; RSA-2048 is the current minimum, RSA-3072 better, RSA-4096 a paranoid default. New designs should generally use elliptic-curve schemes instead. |
| **Weak elliptic curves** | P-192, P-224, or any curve with no formal security analysis. Acceptable: NIST P-256, P-384, P-521; X25519 / Ed25519 (preferred for new designs). secp256k1 acceptable in Bitcoin/blockchain contexts only. |
| **TLS 1.0 / 1.1 accepted** | TLS server configuration permitting protocols below TLS 1.2; or accepting RC4, 3DES, NULL, EXPORT cipher suites. TLS 1.2 is the floor; TLS 1.3 the preferred default for new infrastructure. |
| **Certificate verification disabled** | `verify=False`, `rejectUnauthorized: false`, `InsecureSkipVerify: true`, accepting any certificate. Disables the entire security property of TLS. Look for justifications; if the answer is "tests need this," scope to test code only with a same-line comment. |
| **`DH` / `ECDH` parameter check missing** | Diffie-Hellman or ECDH where the peer's public value is accepted without checking it lies in the expected group / on the curve. Invalid-curve attacks recover the private key. Modern libraries (`X25519`, libsodium) handle this; raw OpenSSL bindings may not. |
| **HMAC implemented as `hash(key + message)`** | Anything that looks like "compute a MAC by hashing the secret prepended to the message." This is the length-extension attack vector (for SHA-256 / SHA-512); HMAC's construction exists specifically to prevent it. Use `hmac` / `crypto.createHmac`. |
| **Truncating an HMAC / signature beyond safe margins** | Cutting an HMAC-SHA256 output to 32 bits (or similar). Reduces forgery resistance proportionally; only safe within RFC-defined truncations (HMAC truncation to ≥ 128 bits is widely accepted). |
| **Encryption of long-lived secrets without PQ consideration** | Backups, archives, signed firmware, encrypted-data-at-rest expected to remain confidential past 2030–2040. New designs should consider hybrid schemes (classical + ML-KEM for key agreement, classical + ML-DSA for signatures) where the secret's relevant lifetime is multi-decade. |

### Dimension 50: Crypto Engineering & Protocol Discipline
*Valsorda, Green*

| Hazard | What to look for |
|--------|-----------------|
| **Custom protocol design** | Code that wires together primitives in a way that constitutes a protocol (key exchange + encryption + authentication, multi-message session, signed-and-encrypted state). Custom protocol design is the highest-risk activity in cryptography; even experts produce CVE-worthy bugs in custom designs. Use Noise, TLS, age, libsodium's `box`/`secretbox`, or an audited high-level scheme. |
| **Primitive composition without an audited template** | Code combining cipher + MAC, KDF + cipher, sign + encrypt by hand, even when each primitive is correct. Encrypt-then-MAC vs MAC-then-Encrypt vs Encrypt-and-MAC produce different security properties; getting the composition wrong is a CVE waiting. Use AEAD (which is the audited composition) or a high-level library. |
| **No domain separation between cryptographic uses of one key** | The same key used to sign multiple message types (login challenges, password-reset tokens, audit logs) with no context-prefix distinguishing them. A signature on one type can be replayed as authentication for another. Prefix signed data with a domain tag (`b"app:auth:v1:"`) or derive sub-keys per use via HKDF. |
| **Algorithm agility without strict allow-list** | Code accepting multiple cryptographic algorithms based on a caller-controlled field (JWT `alg`, content-encoding header, protocol negotiation). Attackers choose the weakest accepted option, or exploit confusion between them. Pin a single algorithm at the boundary, or allow-list with the verifier — never the data — choosing. |
| **JWT `alg` confusion** | JWT verifier that accepts RS256 and HS256 with the same key parameter — attacker submits `alg: HS256` token using the RSA *public* key as the HMAC secret. Pin allowed algorithms explicitly; reject tokens whose algorithm doesn't match the expected one. (Cross-references Identity Dimension 47 — flag here when the issue is the verification-library API, route to Identity when the issue is the OAuth/OIDC integration.) |
| **No forward secrecy for long-lived sessions** | Session establishment using static-key encryption (RSA key transport) rather than ephemeral key agreement (ECDHE, X25519). Recording today's sessions enables future-key-compromise decryption of years of past traffic. TLS 1.3 only supports forward-secret cipher suites; earlier versions need explicit configuration. |
| **Long-lived signing/encryption keys with no rotation plan** | A key generated once, hardcoded or stored, used indefinitely with no documented rotation cadence or revocation mechanism. The longer a key lives, the more value is encrypted under it; the more value, the higher the incentive for a determined attack. Plan rotation; build the rotation path before deploy. |
| **No key revocation / compromise-recovery path** | "What happens if this signing key leaks?" with no answer beyond "we generate a new one." Replacing a long-term verification key requires every relying party to learn the new one — a coordination problem usually unplanned until incident. Use short-lived keys signed by a longer-lived root, or key-transparency mechanisms (CT logs, key directory). |
| **Padding-oracle-shaped error handling** | Decryption code returning distinguishable errors for "invalid padding" vs. "invalid MAC" vs. "wrong key." The distinction is the oracle. Return a single opaque error; constant-time the validation steps. |
| **Cryptographic library defaults trusted without audit** | A library call like `cipher.encrypt(key, plaintext)` where the IV, mode, padding, and tag length are library defaults. Defaults change across library versions — code that worked under v1 silently changes properties under v2. Pin the parameters explicitly. |
| **No known-answer tests for cryptographic code** | Custom implementation (or composition) of a cryptographic primitive with no test vectors against published test cases. The code passes unit tests on data the author generated and fails against the spec it claims to implement. KAT discipline is non-optional. |
| **Cryptographic dependencies unpinned or vendored old** | `package.json` / `Cargo.toml` / `go.mod` allowing minor-version drift on `crypto`-prefixed dependencies; or a vendored copy of a crypto library years behind its source. Pin exact versions; audit upgrades deliberately; subscribe to security advisories for crypto deps. |
| **Plaintext secret in logs** | `logger.debug("decrypted: %s", plaintext)`, `console.log({ key })`, exception traces including key material or unencrypted payload. Telemetry destinations have different access controls than the data; the log is a secret-storage location it shouldn't be. |
| **Key material persisting in memory longer than necessary** | Languages allowing it (Go, Rust, C, C++): key material kept in regular byte slices for the lifetime of the process, no `runtime.KeepAlive` discipline, no `mlock` to prevent swap-to-disk. Memory dumps and core files contain the keys. Zero after use (`crypto/subtle`, `zeroize` crate, `mlock` for sensitive buffers). |
| **Mixing high-level and low-level for the same operation** | Code using `libsodium`'s `crypto_secretbox` for some encryption and raw AES-GCM for other encryption — the inconsistency creates places where review attention is split. Pick one altitude and stay there. |
| **No cryptographic agility for algorithm migration** | An encryption / signing routine called from a hundred places, each hardcoding the algorithm. A future migration (Ed25519 → post-quantum) requires touching every caller. Centralize through one wrapping API; tag stored ciphertexts/signatures with their algorithm version for forward compatibility. |
| **Public-key fingerprint or hash truncated for display** | A "key fingerprint" derived by SHA-256 and then truncated to 8 hex characters for display. Collision rate for visual verification becomes practical; users approve the wrong key. Use longer truncations (≥ 16 hex / 64 bits) or full hashes. |
| **`hmac` used where `kmac` / domain-separated HMAC was needed** | HMAC with the same key used for multiple message types or protocol versions. Re-derives the cross-protocol-reuse risk from Green's domain-separation position. Use HKDF to derive per-use sub-keys, or include a domain tag in the MACed data. |
| **Side-channel surface in branching on secret** | `if secret_byte == expected_byte` or array indexing by secret value within a cryptographic operation. Branch prediction and cache timing leak the secret. Constant-time discipline applies to control flow and memory access patterns, not just comparison utilities. |
| **Cryptographic code merged without specialist review** | A non-trivial cryptographic change merged with general code review only. Cryptographic bugs have higher false-negative cost than most code; specialist review is the right gate. If the team has no specialist, the right answer is usually "don't write this code — use a high-level library." |

Valsorda's design questions: for every encryption operation in this code, is the mode authenticated (AEAD)? If not, who guarantees integrity, and where is the MAC verified? For every random value used as a secret (key, nonce, token, salt), did it come from the CSPRNG? If you can't trace it back to one, find where the entropy enters. For every comparison of a secret to user-supplied data, is it constant-time?

Green's design questions: for every cryptographic key in this code, what protocol uses it, and would that key's compromise compromise other protocols that share it? For every algorithm choice this code makes, can a caller influence it? For every long-lived cryptographic secret (encryption keys for data-at-rest, signing keys, root keys), what's the rotation plan and what's the compromise-recovery plan?

**Severity convention:** Cryptographic primitives misused (Dimension 49) are usually **BLOCKER** when the misuse breaks confidentiality, integrity, or authenticity — even when the immediate impact isn't visible to the application code. Cryptographic engineering issues (Dimension 50) are **BLOCKER** when they constitute a protocol-level flaw (no domain separation, no forward secrecy where required, algorithm confusion enabled), **MAJOR** otherwise. A finding from this panel should be conservative: if the reviewer is not a cryptographer and the issue is subtle, route to a specialist rather than downgrading to OBS.
