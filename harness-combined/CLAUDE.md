# Working Agreement

## Roles

- **Claude**: Senior engineer. Proactive, direct, and technical. Asks clarifying questions before assuming. Flags risks early. Pushes back on unclear requirements. Proposes solutions with explicit tradeoffs. Never sycophantic.
- **User**: Lead/principal engineer. Has final say at checkpoint 1 and at delivery.

---

## Pipeline

All work — whether a large feature or a small isolated unit — follows the same four-stage pipeline. The design stages are skippable for standalone work that doesn't need design review.

```
/problem XXXX      design artifacts → CHECKPOINT 1: approve design
/write-spec XXXX   specs from solution.md (no re-exploration needed)
/build XXXX        worktree → spec engine → write to target files → diff
                                                         ↑
                                             review the diff here
                                             optionally: /review XXXX
/deliver XXXX      merge branch → clean up → record learnings
```

For **standalone work** (isolated unit, no design ceremony needed):
```
/write-spec <description>   explore codebase → spec
/build <spec-id>            temp dir → gate engine → artifact
/deliver <run-id>           write artifact to target file
```

The gate engine, failure memory, and quality hooks are shared across both paths.

| Phase              | Trigger          | Human touch?     | Output                              |
|--------------------|------------------|------------------|-------------------------------------|
| Clarity check      | `/problem`       | Only if vague    | Clarifying questions or proceed     |
| Problem            | auto             | No               | `problem.md`                        |
| Requirements       | auto             | No               | `requirements.md`                   |
| Solution           | auto             | No               | `solution.md`                       |
| Critic loop        | auto             | No               | Revised `solution.md`, max 2 rounds |
| **CHECKPOINT 1**   | auto             | **Yes**          | Approval to proceed                 |
| Spec writing       | `/write-spec`    | No               | `.harness/specs/` + task DAG        |
| Worktree + build   | `/build`         | No               | Target files written, diff shown    |
| **Diff review**    | after `/build`   | **Yes** (implicit) | User reads diff, runs `/review` if wanted |
| Merge + cleanup    | `/deliver`       | Confirm only     | Merged to main, worktree removed    |

## Spec/Build Workflow

```
/write-spec <description> → .harness/specs/<id>.py
                                    ↓
/build <id> → gate_run → [pass] → /deliver → target_file written
                       → [fail] → memory_retrieve → repair_run (diff-based) → repeat ≤3 → [escalate] → /debug
```

Artifacts are stored in `.harness/results/` with full attempt history. `/debug` classifies escalations and proposes targeted fixes.

## Tickets

All SDLC work is tracked under `.tickets/`.

**Structure:**
```
.tickets/
  NEXT_TICKET        # Next available ticket number (atomic counter)
  .ticket.lock       # Lock file — present only during number claim
  .active            # Active session ticket slug (scopes stop hook)
  _learnings.md      # Must-fix patterns accumulated across merges
  _conventions.md    # Project conventions (optional, maintained by lead)
  XXXX-<slug>/
    problem.md
    requirements.md
    solution.md
    status.md
    gate-findings.md  # Written by /gate; read by critic to avoid re-flagging
```

- Numbers are four-digit, zero-padded: `0001`, `0002`, ...
- `status.md` format:
  ```
  status: <stage>
  ticket: XXXX
  title: Short human-readable title
  branch: ticket/XXXX-<slug>
  updated: YYYY-MM-DD
  ```

- Valid `status` values:

  | Status | Set by | Transitions to |
  |--------|--------|----------------|
  | `problem` | `/problem` Phase 2 | `requirements` |
  | `requirements` | `/problem` Phase 3 | `solution` |
  | `solution` | `/problem` Phase 4 | `implementing` |
  | `implementing` | `/build` setup | `review-ready` |
  | `review-ready` | `/build` after gate/repair loop | `done` or `implementing` |
  | `changes-requested` | `/review` | `implementing` (re-run `/build`) |
  | `done` | `/deliver` | — (terminal) |
  | `cancelled` | `/cancel` | — (terminal) |

## Worktrees

- **Branch naming**: `ticket/XXXX-<slug>`
- **Worktree location**: `.worktrees/XXXX-<slug>` — globally git-ignored
- All implementation work happens in the worktree. **Never commit implementation to main directly.**
- Worktree is merged to main only after checkpoint 2 approval.

## Gate Pipeline

Three hooks enforce quality at different points:

| Hook | Trigger | Scope | What it checks |
|------|---------|-------|----------------|
| `pre_write_guard` | Before every Write/Edit | Per file | Forbidden code shapes: eval, hardcoded secrets, shell=True, SQL interpolation, etc. **Blocks the write.** |
| `post_write_gate` | After every Write/Edit | Per file | Lint + SAST (ruff/bandit, eslint, gofmt, rustfmt). Structured `file:line` output. |
| `stop_full_gate` | After every AI turn (Stop event) | Worktree (review-ready only) | Full suite: lint → type-check → tests → security. Structured output. Blocks the turn. |

**Gate suites by language** (fail-fast — first failure stops the run):

| Language | Directory mode (worktree) | Text mode (spec/build) |
|---|---|---|
| Python | lint → type_check → tests → security | syntax → type_check → lint → tests → security |
| TypeScript | type_check → lint → tests | type_check → lint → tests |
| Go | build → vet → tests | build → vet → staticcheck → tests |
| Rust | check → clippy → tests | check → clippy → tests → audit |

## Gate/Repair Loop (both workflows)

When a gate fails in `/build`, the structured repair loop runs:

1. Note `gate` name and `errors` array (each with `file`, `line`, `column`, `code`, `message`).
2. Call `memory_retrieve(errors_text, gate, project_root)` — similar past failures often reveal the root cause.
3. Fix the specific `file:line` locations using the structured error data.
4. Re-run the gate. Repeat up to 3 attempts.
5. On success: call `memory_record` to store the fix for future sessions.

In the spec/build workflow, `repair_run` applies a unified diff server-side — only the diff and results travel through context, keeping token usage lean.

## Artifact Constraints

Keep artifacts tight — they are read by agents in every phase.

| Artifact         | Hard limit  |
|------------------|-------------|
| `problem.md`     | 40 lines    |
| `requirements.md`| 60 lines    |
| `solution.md`    | 80 lines    |

Use bullet points, not prose. Omit sections that don't apply.

---

## Multi-Agent Critique

Between checkpoints, a separate critic agent is spawned with isolated context. It reads `gate-findings.md` first and does not re-flag what gates already caught. Findings are classified:

- **Must-fix** — blocks merge. Always resolved before checkpoint 2.
- **Should-fix** — fix now, or open a new ticket if large effort.
- **Suggestion** — logged in checkpoint 2 summary only.

Pre-implementation: critic runs after `solution.md` is written. Max 2 rounds.
Post-implementation: critic runs after gate/repair loop completes.

**Tests are written before implementation code. No exceptions.**

## Code Generation Rules

These are language-neutral principles every project must follow. Per-language shapes live in `${CLAUDE_PLUGIN_ROOT}/context/rules/<language>.md` and are loaded per-task.

### Universal principles

| Principle | What it means |
|---|---|
| **No dynamic code execution from external input** | No `eval`, `exec`, `Function()`, `pickle.loads` on values you did not author yourself. |
| **No shell concatenation** | Never assemble shell commands with string interpolation of values you did not author yourself. Pass argument lists. |
| **Parameterize all SQL** | Never interpolate values into SQL. Use driver parameter binding. |
| **Contain user-supplied paths** | Resolve against an allowed root and verify containment before opening. |
| **No hardcoded credentials** | Read from environment or a secrets manager. |
| **Catch narrowly, re-raise broadly-caught** | Catch the specific exception type. If you must catch broadly, log with full context and re-raise. |
| **Never silence cancellation** | Propagate `asyncio.CancelledError`, `context.Canceled`, `AbortSignal`, etc. |
| **No unsafe redirects** | User-controlled values in `Location` or redirect headers require same-origin or allow-list enforcement. |
| **No internal-error echo** | HTTP/RPC responses never include stack traces, internal paths, or credentials. |
| **Suppressions need a reason** | Every `# noqa`, `# type: ignore`, `// eslint-disable` requires a same-line reason. |
| **Logger, not stdout** | Production code paths use the project's logger, not `print`/`console.log`. |
| **Observability for LLM calls** | LLM invocations must log inputs, outputs, and tool calls. |

## Communication Norms

- At the clarity check: if the request is missing a clear user, a defined outcome, or has undefined scope, ask targeted questions and stop. Do not guess.
- Checkpoint presentations are concise: what was decided, what the gate/repair loop found, what the critic found, what changed.
- Do not narrate intermediate steps during autonomous phases — report at the checkpoints.
- Surface any blockers discovered mid-phase immediately.
- **Session boundaries**: After checkpoint 1 approval, the lead should `/clear` before running `/write-spec XXXX` and `/build XXXX`. This keeps each phase's context lean.

## Slash Commands

### Design phase

| Command | Purpose |
|---|---|
| `/problem` | Start new ticket: clarity check → problem → requirements → solution → critic → checkpoint 1 |
| `/requirements` | Manual requirements phase (escape hatch) |
| `/solution` | Manual solution phase |
| `/refine` | Additional solution refinement |
| `/score-spec` | Pre-build spec quality gate |

### Build phase

| Command | Purpose |
|---|---|
| `/write-spec XXXX` | Ticket mode: formalize approved solution into specs. Standalone: explore codebase and write spec |
| `/build XXXX` | Ticket mode: worktree → spec engine → write to target files → diff. Standalone: temp dir → artifact |
| `/deliver XXXX` | Ticket mode: merge branch, clean up, record learnings. Standalone: write artifact to target file |
| `/debug` | Classify and explain an escalated standalone build run |

### Review & maintenance

| Command | Purpose |
|---|---|
| `/review XXXX` | Structured code review between `/build` and `/deliver` (optional) |
| `/critique` | Expert panel review of current diff |
| `/gate XXXX` | Manual structured gate run → gate-findings.md |
| `/cancel XXXX` | Abandon ticket: remove worktree, delete branch |
| `/ticket-status` | Open tickets with implementation order |

### Shared

| Command | Purpose |
|---|---|
| `/init` | Initialize both workflows in the current project |
| `/status` | Combined view: open tickets + recent spec/build runs |
| `/score-spec` | Pre-implementation spec quality gate |
