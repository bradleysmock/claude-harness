## Findings

**MINOR** — Core / Dimension 4 (Documentation & Comments) — `context/flows/deliver-ticket.md:253-269`

Step 1.7 (lines 70-85) states the marker's contents must be printed "as part of the Step 3 confirmation block" when present. But Step 3's own confirmation-block template (lines 253-269) has no placeholder line for it, unlike the way the same template explicitly handles other conditional content — the bracketed `[Note: gates ran on...]` (rebase), `[git push origin <branch>]` / `[gh pr create ...]` (`--pr`), and `[smoke test: ...]` (smoke-test config) lines all show exactly where and how they're rendered. A reader implementing Step 3 from the template alone would not know where the marker contents belong in the printed block. Fix: add a bracketed `[refine-touched.md contents — only when marker present]` line to the template, mirroring the other conditional lines.

**MINOR** — Core / Dimension 9 (Domain Modeling) + NFR-1 (fail-closed) — `context/spec-remediation.md:81-118`, `ticket.py:674-698`

The marker's lifecycle has no "retract" path outside of `_fold_archive` at delivery. Scenario: S2 writes+commits `refine-touched.md`, then `/refine` hits its own bail condition (cannot derive the fix from existing text) — S2 goes to Hard-stop and returns `bail` with the marker already committed on the branch. If the lead then manually edits `requirements.md`/`solution.md` and re-runs `/autopilot XXXX`, and this time the mechanical pass alone clears the BLOCK (`succeeded(autonomous=True)`, S2 never re-entered), the stale marker from the earlier abandoned attempt is never cleared — nothing in S0/S1's `succeeded(autonomous=True)` path deletes a pre-existing marker. Step B / deliver will then force an unnecessary confirmation and display an audit trail describing a `/refine` pass whose changes are not part of the final accepted scope. This doesn't violate fail-closed correctness (it over-triggers confirmation rather than under-triggering it), so it's not a blocker, but it's an unaddressed edge case in both requirements.md and solution.md's Tradeoffs section (which discusses "double confirmation" for the Step-B-then-direct-`/deliver` case but not this stale-marker-from-abandoned-attempt case).
