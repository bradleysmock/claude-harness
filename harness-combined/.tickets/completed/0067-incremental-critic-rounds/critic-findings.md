**MINOR** · Python / Dimension 10 (Pythonic design) & Testing / Dimension 22 (test strategy — untested edge case) · `gates/incremental_scope.py:101-117` <!-- harness-finding-key :None:MINOR:Python / Dimension 10 (Pythonic design) & Testing / Dimension 22 (test strategy — untested edge case) -->

`touched_files_from_diff`'s `--- `/`+++ ` matching has no hunk-boundary awareness — it pattern-matches any line starting with `--- ` or `+++ `, including lines *inside* a hunk body, not just the file-header pair at the top of each file's diff block. A deleted line whose original content begins with `-- ` (two dashes + space — a common comment prefix in SQL, Lua, Haskell, AppleScript) becomes, after git's single `-` hunk marker is prepended, a line reading `--- <rest>`, which false-matches the file-header regex and corrupts `old_path`/spuriously adds a bogus entry to `touched`. The analogous case (`++ ` content on an added line) false-matches `+++ `. Trace: this cannot cause a *real* touched file to be dropped (the genuine header pair always precedes the hunk body for that file), but it can inject spurious non-existent paths into the touched-files list handed to `panel_detect.py`. None of the five test fixtures (`MODIFY_DIFF`, `RENAME_DIFF`, `BINARY_DIFF`, `DELETE_DIFF`, the escaping-path diff) exercise a hunk body whose content starts with `-- `/`++ `, so this is untested. **Fix:** track hunk-body state explicitly (reset on `diff --git`, set on `@@ ... @@`, and only treat `--- `/`+++ ` lines as file headers when *not* currently inside a hunk body), or restrict the header match to lines immediately following an `index `/`diff --git` line.

**MINOR** · Core / Dimension 1 (Abstraction Quality — interface completeness) · `gates/incremental_scope.py:132-164` <!-- harness-finding-key :None:MINOR:Core / Dimension 1 (Abstraction Quality — interface completeness) -->

`format_incremental_brief`'s docstring ("the brief text itself") and the module's own top-level docstring ("`format_incremental_brief` (the brief text itself)") describe it as producing the incremental round's complete brief, but it only renders two of the three pieces a round-N+1 spawn prompt needs — the prior findings and the diff. The touched-files list (`touched_files_from_diff`'s own separate output) is *not* embedded by this function; `build-ticket.md` Step 7a item 5 has to separately splice it into the prompt ("spawn it with `Mode: incremental`, the touched-files list, and the formatted brief from item 4b"). A future caller reading only this module's docstring, without cross-referencing `build-ticket.md`, could reasonably assume calling `format_incremental_brief` alone produces a complete, spawn-ready brief and omit the touched-files list from the prompt — silently breaking Step 1's diff-scoped panel loading. **Fix:** either fold the touched-files list into `format_incremental_brief`'s output (accepting `touched_files: list[str]` as a third parameter) or rename/re-document the function to make explicit that it covers only the findings+diff sections, not the full spawn payload.

`_contained(path, worktree_root) -> str | None` reads as a boolean predicate (`_contained` = "is contained") but returns the validated path or `None`, not a bool — callers do `contained = _contained(...); if contained: touched.add(contained)`, which works but the name doesn't signal the return-value contract.
